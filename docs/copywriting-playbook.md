# GitHub 文案工程手册（Copywriting Playbook）

本手册用于保证：每次功能推进后，README 都能同步反映真实状态，并持续吸引贡献者加入。

## 核心原则

1. 真实优先（Reality First）
- 只写已经可运行、可验证的能力
- 规划项必须明确标注为“规划中”

2. 用户价值优先（Value First）
- 每次更新都要解释“这对用户有什么帮助”
- 尤其说明对老年人、肢体不便用户的潜在价值

3. 可执行优先（Actionable）
- 文案里必须有可复制命令
- 至少保留一条“最快跑通路径”

4. 一致性优先（Consistency）
- README、CONTRIBUTING、ROADMAP 不冲突
- 版本日期、功能状态、命令示例必须同步

## 每次 Push 前文案检查清单

- [ ] README 的 “当前状态（As of YYYY-MM-DD）” 已更新
- [ ] “已实现 / 规划中” 分层准确
- [ ] 新功能有用户价值描述，不只写技术细节
- [ ] 至少新增 1 条真实命令示例
- [ ] 新依赖和运行前提已补全
- [ ] 安全边界变化已更新
- [ ] 贡献者入口（Issue/PR 指引）仍清晰

## 推荐结构模板（README）

1. 项目一句话定位（Desktop Pet Assistant）
2. 项目愿景（Chat-to-Action / Assistive Computing）
3. 当前状态（日期 + 能力快照）
4. 能力矩阵（已实现 vs 规划中）
5. 快速开始 + 测试命令
6. 安全边界
7. 贡献方向与行动入口

## 文案语气建议

- 中文为主，关键术语可附英文关键词（如 Desktop Pet, Skills, Accessibility）
- 避免夸张营销词，强调“真实可运行”
- 保持短句、可扫描、可复制

## 禁止项

- 不允许把未实现能力写成“已支持”
- 不允许省略安全边界说明
- 不允许只改代码不改 README（当功能发生变化时）
